## Glossary

**Waterline:** A bar chart with one bar per team showing allocated story points for a release,
with a horizontal threshold line marking the fill limit. Bars change colour dynamically as ideas
are added, removed, resized, or reassigned: below threshold = headroom (green); between threshold
and full capacity = filling/caution (amber); above capacity = over-allocated (red).

**Idea:** In planning mode, the atomic unit of work. During the ROAD-to-JPD transition, an idea
is either a JPD idea or a ROAD item (see ROAD definition below). After migration completes, all
ideas are JPD ideas. The T-shirt size and team assignment live on the idea; the Jira Version link
connects it to a release.

**ROAD:** The current custom Jira project (key: to be confirmed on sterlandsupport.atlassian.net)
used for high-level planning before JPD adoption. ROAD items are standard Jira issues. The app
reads their T-shirt size from the custom field used today for sizing (field key to be confirmed
during the JPD spike in The Assignment). After full JPD migration, ROAD items will no longer
appear in the waterline. The app must handle both sources during the transition period.

**Threshold:** A configurable percentage of team capacity, stored per release in Forge Storage
(default: 70%). Rendered as a horizontal line on the waterline bar chart, distinct from the
full-capacity bar top. An engineer can change the default in app config; a DM can override it
per release.

## Problem Statement

Jira does not handle release-level or sprint-level team capacity planning. The result is a
spreadsheet that is **retrospective** — it captures what already happened, not what is planned.
Because no one can see the capacity envelope before scope is committed, releases get overloaded,
priorities shift mid-sprint, and scope change is managed by gut feel rather than visible data.

**The core temporal problem:** the value is not better data — it is data that arrives *before*
commitments are made. The tool must be open during the planning meeting, not reviewed after.

## Status Quo

The product team creates high-level plans with no estimates — allocation is gut feel based on
experience. Visibility is limited to active and next sprint only. Priorities shift constantly as
capacity changes. The spreadsheet is retrospective — it confirms overload after it has already
happened. The transition from ROAD to JPD is in progress, making now the right time to establish
a proper capacity layer.

## Target User & Narrowest Wedge

**Primary:** Delivery Manager — runs release planning, owns the capacity numbers, needs the
waterline open during the planning meeting to stop scope from being committed before the envelope
is checked.

**Secondary:** Head of Product, all product owners — make allocation decisions without capacity
data today.

**Narrowest wedge (planning mode):** A live waterline for one release — team capacity bars,
T-shirt sized ideas, a visible threshold line — open during a planning meeting.

## Constraints

- Platform: Atlassian Forge (cloud-native; data stays in Atlassian cloud).
- Instance: sterlandsupport.atlassian.net (Cloud ID 27fae7eb-3b1d-4aa0-9a87-d8a37082592e).
- JPD adoption is in progress — planning mode must work with both ROAD items and JPD ideas
  during the transition (see Glossary — ROAD).
- Jira Version is the anchor object for a release. Version tagging discipline is being enforced
  as part of this rollout.
- T-shirt sizes live on JPD ideas as a JPD-native story-points field (or custom field — exact
  field key to be confirmed in the JPD spike; see The Assignment). During ROAD transition, the
  same custom field used on ROAD items today will be read instead.
- T-shirt → story points: configurable map, stored in Forge Storage (default: XS=1, S=3, M=8,
  L=13, XL=21). No per-person availability modelling in v1.
- Capacity numbers: the DM enters the effective team capacity for the release (story points /
  velocity units), reduced from raw team velocity to account for planned leave, overhead, or
  other known constraints. A free-text note field accompanies each entry.
- Waterline re-colouring must feel live: client-side recompute on every change, not a page reload or Forge round-trip.
- **Write model (two paths):** Team assignment writes go directly to Jira asynchronously on each reassignment (no Save button for this action). Capacity numbers and threshold settings are saved to Forge Storage only on explicit user action via a Save button; unsaved changes show an indicator. Concurrent writes to Forge Storage are last-write-wins. Two users editing simultaneously is not supported in v1; the app is designed for a single editor (DM) during a planning session.
- **Access control:** Capacity and threshold editing is restricted to Jira project admins or a configurable set of named users, managed via the Forge app configuration page (accessible to Jira site admins only; input: Atlassian account IDs). Read access is open to all project members. Idea T-shirt sizing and team assignment use standard Jira/JPD write permissions.
- **Forge Storage schema — releases** (one key per release): `Key: "release:<versionId>"` → `{ teams: { [teamId]: { capacity: number } }, threshold: number, warningThresholdPct: number }`. `threshold` is the fill-target percentage (default 70). `warningThresholdPct` is the version-tagging coverage threshold below which the "ideas may be missing" banner appears (default 10% untagged = 90% coverage required; configurable per release). Note field on team capacity deliberately omitted from v1 schema — deferred to v1.1.
- **Forge Storage schema — teams** (one key per install): `Key: "teams"` → `{ teams: [{ id: string, name: string, boardId: number }] }`. Managed via the Forge app configuration page (site-admin only). Team IDs are stable internal identifiers; `boardId` links a team to a Jira board for delivery-mode sprint queries. Board IDs are optional in v1 (planning mode does not use them); required before delivery mode begins.
- **Forge Storage partitioning:** One key per release, partitioned to stay within the 32 KB per-value limit. Quota validation: confirm key limit at steady state (projected releases/year × 3-year retention < 80% of documented Forge Storage key quota) during Spike #2.
- **Epic traversal algorithm:** Query for all Epics with `fixVersion = <selectedVersion>` via Forge API. Fetch child issues of those Epics in the same Forge function call, handling Jira REST API pagination internally (max 100 results per page; loop until `startAt + maxResults >= total`). Return the full aggregated payload to the React frontend as a single object. Note: for large releases this loop runs inside the Forge function — latency scales with issue count. The Spike #1 load test must use a real release with representative volume (not a synthetic empty version) to validate the < 1.5 s target holds under pagination.
- No external reporting tools; no admins to operate it.
- Planned Version lives at Epic level — any release-assembly traversal reads version via the parent Epic, not the Story.

## Error States

| State | User sees |
| --- | --- |
| Version has zero tagged ideas | Empty-state message: "No ideas linked to this version. Add ideas in JPD or link existing issues." |
| A team has no capacity entered | Bar shows 0 with an "Enter capacity" prompt inline. Threshold line not drawn until capacity is set. |
| Forge API call fails on page load | Full-page error with a Retry button. Partial failure (one team's data missing) shows that team's bar in an error state with a tooltip. |
| Idea has no T-shirt size set | Idea appears in the list with a "—" size indicator; it contributes 0 points to the waterline. A count of unsized ideas is shown above the waterline. |
| Idea has no team assigned | Idea appears in an "Unassigned" bucket below all team bars; contributes 0 points to any team waterline. Count of unassigned ideas shown alongside unsized count. |
| Version tagging rate below configured threshold | Warning banner: "N ideas have no version tag — waterline may be incomplete." Denominator: count of open JPD/ROAD ideas in active planning status with no fixVersion set. Threshold (default: show banner when untagged count > 10% of in-version ideas) is configurable. |
| Async Jira write fails (team reassignment) | Non-blocking toast: "Reassignment could not be saved. Retry?" In-memory state reverts to prior team assignment if user does not retry. |

## Premises

1. The DM + Head of Product + POs will use it — demand is pre-validated by daily process pain.
2. Forge is the right platform: data stays in Atlassian cloud, integrates natively with JPD and Jira Versions, no external infra required.
3. T-shirt → story points model is sufficient for planning mode. Per-person leave scheduling is out of scope for v1.
4. Jira Version is the right anchor object for a release. Version tagging discipline is being enforced as part of this rollout — the tool reinforces that discipline. **Caveat:** If version tagging compliance falls below ~70% of in-flight ideas, the waterline data will be unreliable. Measure tagging rate before launch and surface it to users (see Error States).
5. The first deliverable is a live planning-mode waterline for one real release — something the Head of Product can open in a browser at a planning meeting.

## Design Risks

**Forge Custom UI bridge latency:** Every call from the React frontend to Forge Storage or the
Jira API crosses an async bridge that adds 1–3 seconds per call. The mitigation is to load all
data in a single Forge function call on page open, keep all state in React, compute the waterline
client-side on every change, and persist back to Forge Storage only on explicit save. The latency
spike (see The Assignment) must confirm this is achievable before application logic is written.

**Write-back latency on idea reassignment:** Reassigning an idea between teams requires a write
call to Jira (updating the idea's team assignment field). This adds a second Forge round-trip on
top of the read path. The latency spike must measure this separately.

**JPD T-shirt field via Forge API — potential hard blocker:** JPD-native fields are not always
exposed as standard Jira custom fields via REST v3. If the Forge API cannot read T-shirt sizes
from JPD ideas, the core planning-mode data model breaks entirely. This must be verified before
any application logic is written (see The Assignment — Spike #2).

**Sprint-to-board data model for delivery mode:** Jira sprints are owned by a board, not a
project or version. There is no "sprints associated with a version" API endpoint. Linking sprints
from four team boards to a single release requires iterating boards and cross-referencing issue
version tags — potentially dozens of API calls. This may make delivery mode latency incompatible
with client-side recompute. Delivery mode must begin with a data model proposal (e.g., manually
curated sprint-to-release mapping in Forge Storage) before any implementation, not after.

## Approaches Considered

### Approach A: Planning Mode Only — Forge Custom UI

- Build just the planning-mode waterline (requirements P1–P10 from the spec).
- Skip delivery mode entirely for v1.
- Effort: M (~3–5 weeks solo). Risk: Low-Medium.

### Approach B: Full Spec — Both Modes

- Implement planning + delivery mode as spec'd.
- Effort: XL (~3–4 months solo). Risk: High.

### Approach C: Next.js Prototype → Validate → Decide

- Standalone Next.js page (Vercel) in 1–2 weeks using Jira REST API directly.
- Effort: XS prototype + decision point + L Forge rebuild if validated. Risk: Low → Medium.

---

## Recommended Approach

**Phased: A then B (builder's choice)**

v1 scope = Approach A (planning mode only). Approach B (delivery mode) is the committed roadmap
once planning mode is validated in a real planning meeting.

Rationale: planning mode closes the primary pain (the "temporal" gap — data during the meeting,
not after). Delivery mode can follow once planning mode is validated. Building both modes in
sequence rather than in parallel reduces risk.

#### Effort estimates

- Planning mode (v1): M, ~3–5 weeks solo.
- Delivery mode (v2): L–XL, ~2–3 additional months. Requires data model proposal first (see Design Risks — Sprint-to-board).

  bars rather than gut feel.
- The spreadsheet is retired (no longer updated as the source of truth for release planning).

## The Assignment

**Complete two spikes before writing a line of application logic.** Both spikes are prerequisite
gates; skip either and you risk building on a broken foundation.

### Spike #1 — Forge bridge latency

Build one Forge Custom UI page that:
1. Calls a Forge function to fetch all issues for a real Jira Version from sterlandsupport.
2. Returns them to the React frontend as a single payload.
3. Renders a bar chart (one bar per team, story points summed) with a hardcoded capacity line.
4. **Also:** performs a single issue field update (team assignment) to simulate a reassignment write and measures the round-trip time.

**Pass criteria:** (a) page-open-to-chart < 1.5 seconds; (b) client-side recompute after reassignment < 200 ms. Jira write round-trip measured for observability (target: completes within 5 seconds) but not a blocking gate — it is async and must not block the UI.

**If fail on (a):** redesign the fetch strategy (single Forge function call returning all data as one payload; all waterline logic in React state) before proceeding.

### Spike #2 — JPD field readability

1. Fetch one JPD idea via the Forge API and confirm the T-shirt/story-size field is readable and its API field key.
2. Fetch one ROAD item from the current ROAD project and confirm the same (or equivalent) field is readable.
3. Confirm the Jira/JPD field used for team assignment and verify it is writable via the Forge API.
4. Confirm the Forge manifest scopes required for read access to JPD ideas and write access to the team assignment field.
5. Confirm Forge Storage key limit; verify that estimated steady-state key count (projected releases/year × 3-year retention) is below 80% of the documented quota.
6. Record: JPD field key, ROAD project key, ROAD sizing field key, team assignment field key. These become constants in the app configuration.

**Pass criteria:** all field keys confirmed readable/writable, manifest scopes confirmed, Forge Storage quota validated.

**If fail on T-shirt fields:** the planning-mode data model requires a redesign — DM enters T-shirt sizes manually in Forge Storage rather than reading from Jira/JPD. Estimated additional effort: S (~1 week). Decision to pivot requires DM sign-off before any further work proceeds.

**If fail on team assignment field:** team reassignment must be implemented via a different writable field (e.g., a JPD custom field or a Jira label). Resolve before writing any reassignment logic.
